# AGENTS.md

## 通用响应规则

- Always respond in Chinese-simplified.

## 工作流程规则

涉及开发、脚本、自动化工具、文档方案落地时，必须按以下顺序执行：

1. 必须先规划。
2. 然后编写或修改 RPD。
3. 然后生成或修改 todo。
4. 然后进行 git 提交，做好回滚准备。
5. 最后才执行实现或批处理任务。

若用户明确要求某一步等待审核，则必须在该审核点暂停，不得继续执行后续实现。

## 工业级稳定规则

最终结果稳定可靠是本项目最高优先级。

当稳定可靠与速度、自动化程度、少走步骤发生冲突时，必须优先选择稳定可靠。涉及机械图纸方向识别、标题栏检测、OCR/VLM 兜底、YOLO/OBB 训练、PDF 校正或批量处理时，不得为了推进进度跳过人工复核、可视化检查、数据校验、错误分层或回滚准备。

## 人工填写与审核界面规则

所有需要用户填写的表格，只显示用户完成当前判断必须看到的信息。

表格中不得暴露用户无需看到的内部字段、调试分数、冗余路径、长 JSON、候选列表或算法细节。技术字段应放入单独 JSON、机器报告或日志文件中，人工填写表只保留低噪声、可直接操作的字段。

所有需要用户审核图片的 HTML 页面，图片排列必须优先考虑人工审核是否方便。

审核页面应保证图纸主体和标题栏足够清晰，按稳定顺序排列，减少无关文字和调试信息；对比或复核任务应让相关图片靠近展示，并使用适合快速浏览、逐张放大和避免裁切的布局。

## 固定审核入口规则

所有需要用户审核、填写或标注的当前任务文件，必须统一放在固定入口：

```text
local_data/review_inbox/current/
```

不得让用户到多个业务目录、输出目录或脚本目录中寻找当前要审核的文件。

当前审核入口只放本轮需要用户直接打开、填写或参考的文件，并提供简短说明。机器清单、调试报告和中间产物可以随任务生成，但不得作为用户入口。

用户审核完成后，必须由自动化流程或执行者将 `current` 中的文件归档或移动到对应业务目录，并记录归档位置；随后下一轮审核再重新生成 `current`。

# OCR + 图像识别大模型兜底流程规划

## 背景

当前 OpenCV 在原始人工确认集和 90 度增强集上的联合评估结果为 83/83，复核数为 0。但未来处理新 PDF 时可能遇到扫描更差、标题栏版式不同、表格干扰更强的页面。

用户希望加入 OCR 和图像识别大模型，例如智谱、阿里视觉模型。核心问题是：串行工作流，还是并行识别后投票。

## 结论

采用：

```text
OpenCV 主流程 + 条件并行兜底 + 证据融合
```

不采用：

- 无条件全量串行。
- 无条件全量并行投票。
- 大模型直接替代 OpenCV 主流程。

## 对 OCR 与 VLM 等价性的修正

OCR 与 VLM 不应等价看待，也不应默认权重相同。

- OCR 擅长提供文字证据，例如是否识别到图号、日期、材料、审核、公司名等标题栏关键词。
- OCR 不擅长稳定理解复杂图纸布局，也不一定能可靠判断标题栏整体位置。
- VLM 擅长整体视觉理解和非典型布局判断，但输出可能受提示词、供应商版本和模型自信度影响。
- 因此 OCR 是“文字证据分支”，VLM 是“视觉语义分支”，两者不是同一种票。

如果要增加第三类识别手段，优先考虑：

1. 专用标题栏检测模型，例如 YOLO/OBB title block detector。
2. 专用 4 类方向分类器，例如轻量 CNN/ViT rotation classifier。
3. 几何/投影方向检测，例如 Hough、Radon、投影直方图、图框与标题栏模板匹配。

这些手段比 OCR 更接近“方向/位置识别器”，也比通用 VLM 更可控。

## 推荐流程

1. OpenCV 先跑，输出标题栏位置、旋转角度、置信度、候选证据和 debug 图。
2. 质量门判断：
   - 高置信且无冲突：直接接受。
   - 低置信、候选竞争激烈、规则冲突、抽检命中：进入兜底。
3. 兜底阶段按需并行运行：
   - OCR：识别标题栏关键词、文字方向、文字框位置。
   - 专用检测/分类器：检测标题栏框或直接预测 0/90/180/270。
   - VLM：判断当前屏幕坐标下标题栏位置，输出结构化证据。
4. 证据融合：
   - OpenCV 几何证据为主。
   - OCR 作为文字和标题栏关键词证据。
   - VLM 作为全局视觉理解证据。
5. 若融合后仍冲突，进入人工复核。

## 触发条件草案

满足任一条件时触发 OCR/VLM 兜底：

- OpenCV `confidence < 0.35`。
- 第一候选与第二候选分差过小。
- 侧边带评分与局部候选方向冲突。
- 页面来自新供应商、新图幅或新扫描批次。
- 批量任务抽检命中。
- 用户指定强制多模型复核。

## Provider 输出协议

所有识别器必须输出统一结构：

```json
{
  "provider": "ocr",
  "title_block_position": "right_top",
  "rotation_degrees": 270,
  "confidence": 0.78,
  "evidence": [
    "识别到图号字段",
    "识别到日期字段",
    "关键词集中在右上方"
  ],
  "raw_output_path": "outputs/..."
}
```

## 融合策略草案

不做简单多数票。建议采用加权证据：

| 证据 | 适合判断 | 默认权重 |
| --- | --- | --- |
| OpenCV 几何结构 | 标题栏候选位置、表格形态 | 高 |
| OCR 关键词 | 标题栏真实性、文字方向 | 中 |
| 专用标题栏/方向模型 | 标题栏框、方向分类 | 中高 |
| VLM 视觉理解 | 非典型布局、全局判断 | 中 |
| 人工确认 | 最终真值 | 最高 |

若 OpenCV 与 OCR/专用模型/VLM 冲突：

- OpenCV 高置信且 OCR/VLM 无明确证据：保持 OpenCV。
- OpenCV 低置信，OCR 和 VLM 一致：采纳 OCR/VLM，标记融合来源。
- 专用标题栏检测模型与 OpenCV 几何证据一致时，可优先于 OCR 文本证据。
- 多方冲突：人工复核。

## 风险

- OCR 对浅色扫描和旋转文字可能不稳定。
- VLM 的自报置信度不一定可校准。
- 外部 API 涉及图纸数据外发，必须默认最小化调用。
- 供应商模型版本变化可能导致结果漂移。
